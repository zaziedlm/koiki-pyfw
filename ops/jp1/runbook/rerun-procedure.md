# JP1運用向け リラン手順書（標準）

## 0. 目的

JP1運用下でバッチが失敗した際に、再実行可否の判断と安全なリラン手順を標準化する。

## 1. 対象

- `ops/jp1/scripts/run-job.sh` 経由で起動される全ジョブ
- 終了コード `20`（業務エラー）または `30`（システムエラー）を返したジョブ

## 2. 事前確認（必須）

1. JP1 実行ID・ジョブ名・実行時刻を記録する。
2. 実行ログ (`JP1_RC` 出力行) を確認する。
3. 失敗分類を確定する（業務 / システム）。
4. ジョブのリラン方式を確認する。
   - A: restart（途中再開）
   - B: re-run（先頭から再実行）
   - C: rerun禁止（手動復旧のみ）

## 3. 失敗分類ごとの対応

### 3.1 業務エラー（Code=20）

- 代表例: 必須データ不足、業務日付不整合、マスタ未整備
- 対応:
  1. 業務部門へ不整合内容を連絡
  2. データ補正完了の証跡を取得
  3. 同一パラメータでリラン

### 3.2 システムエラー（Code=30）

- 代表例: DB接続断、タイムアウト、ディスク不足、依存API障害
- 対応:
  1. 基盤要因を復旧（DB/ネットワーク/ミドル）
  2. 復旧確認（疎通、容量、接続）
  3. リラン

## 4. リラン実行手順（標準コマンド）

```bash
# 例: 本番で customer-daily-sync を 20260519 分で再実行
./ops/jp1/scripts/run-job.sh \
  -j customer-daily-sync \
  -e prd \
  -a /opt/koiki/apps/customer-a-batch-app.jar \
  -p 20260519 \
  -r JP1-RERUN-20260519-001
```

- `-r` は必ず新しいリクエストIDを付与する。
- `-p` は原則、初回実行と同一業務日付を指定する。

## 5. リラン後の確認

1. 終了コードが `0` または運用合意済み警告コード `10` であること。
2. 件数整合（入力件数/処理件数/エラー件数）を確認する。
3. 下流連携ファイルやDB更新の重複がないことを確認する。
4. JP1ジョブネットが後続まで正常進行したことを確認する。

## 6. エスカレーション条件

以下に該当したら、同日中に運用責任者へエスカレーション。

- 同一ジョブが2回連続で `30` 終了
- `20` の原因が業務側で即日解消できない
- リランにより二重計上・重複連携リスクがある

## 7. 禁止事項

- 原因未確定のまま無制限リランを繰り返すこと
- 手順書外のパラメータで本番実行すること
- 失敗ログを採取せずに復旧完了扱いにすること
