# SAML 連携における業務 `user` テーブル採用の考え方

## この資料で伝えたいこと

今回の構成では、業務アプリ開発チームが SAML の内部仕様や SSO 専用テーブル設計を新たに考えなくても、
業務側の `user` テーブルに用意済みの一部カラムを SSO 連携先として採用するだけで、
既存フレームワークの認証機能と連動できます。

開発メンバーが合意すべきことは、基本的に次の 2 点です。

1. 業務側 `user` テーブルのどの列を SSO 連携に使うか
2. その列を業務アプリとして正式に採用してよいか

今回の検証では、この前提はすでに満たせています。  
そのため、業務アプリ側は「SAML の仕組みを再実装する」のではなく、
「業務 `user` テーブルの対象カラムを SSO 連携の保存先として採用する」と決めれば足ります。

## 業務アプリ側で新たに考えなくてよいこと

- `user_sso` テーブルを新設すること
- SAML Assertion の解析処理を業務アプリ側で実装すること
- フレームワーク認証との橋渡しを業務アプリ側で個別に作ること
- SSO 成功後のトークン発行フローを業務アプリ側で組み直すこと

これらは、今回の改修でフレームワーク側に吸収されています。

## 業務アプリ側で決めればよいこと

- `SSO_LINK_BACKEND=user_table` を採用するか
- 業務 `user` テーブルの SSO 用列を正式運用の保存先として認めるか
- 必要なら、業務上の固定値をどの列に入れるか

## 今回の採用方針

今回の `user` テーブルでは、以下の列を SSO 連携に利用します。

| 用途 | 採用する業務 `user` テーブル列 |
|---|---|
| SSO プロバイダ識別子 | `user_sso_provider` |
| SSO ユーザー識別子 | `user_sso_subject` |
| SSO 連携メール | `user_email` |
| SSO 連携表示名 | `user_name` |
| 最終 SSO ログイン時刻 | `updated_at` |
| 有効フラグ | `user_is_active` |
| 論理削除判定 | `is_deleted` |

この構成により、今回必要な情報は過不足なく充足しています。  
つまり、業務アプリ側では追加の SSO 専用テーブル設計を前提にしなくても、
この `user` テーブルで SSO 連携を成立させられます。

## 全体像

```text
IdP (SAML)
   |
   | SAML 認証成功
   v
フレームワーク SAML 処理
   |
   | user_table backend を使用
   v
業務 user テーブル
  - user_sso_provider
  - user_sso_subject
  - user_email
  - user_name
  - updated_at
   |
   | 連携済みユーザーとして扱う
   v
既存フレームワーク認証
   |
   | アクセストークン / リフレッシュトークン発行
   v
業務アプリ画面
```

## アプリ開発メンバー向けの理解ポイント

- SAML 認証そのものはフレームワーク側が処理する
- 業務アプリ側は「どの `user` 列を SSO 情報の保存先にするか」を理解すればよい
- 今回は必要列がすでにあるため、追加設計の中心は少ない
- そのため、設計レビューでは SAML の理論よりも `user` テーブルの採用方針を揃えることが重要

## 実装上の意味

- `SSO_LINK_BACKEND=user_table` の設定で、標準 `user_sso` ではなく業務 `user` テーブルを利用する
- SAML ログイン成功時、必要な SSO 情報が業務 `user` テーブルへ保存される
- その後は既存のフレームワーク認証に引き継がれる
- 結果として、業務アプリは SSO 対応のための大きな追加実装を持たずに済む
