# Backend Enterprise Audit

最終更新: 2026-04-08

## 1. この文書の目的

この文書は、`koiki-pyfw` の現行コードベースを Enterprise 品質の観点で点検した結果を、
今後の修正作業計画に落とし込みやすい形で整理したものです。

個別バグの網羅列挙ではなく、対策テーマごとに以下を明確にすることを目的とします。

- 何が問題か
- なぜ Enterprise 品質上の課題なのか
- どの状態まで直せば完了と言えるか
- どの領域に変更が波及するか
- どの観点で検証すべきか

また、既存の点検レポートに記載があった論点かどうかも併記します。

## 2. 総合評価

現状は、レイヤード構造や認証基盤、SSO/SAML 拡張など、Enterprise 向けの骨格は整っています。
一方で、本番運用を支えるために重要な `設定統制`、`トランザクション整合性`、`監査/監視`、
`CI による品質保証` が十分に確立されておらず、現時点では「Enterprise 品質で安定運用可能」とは評価しにくい状態です。

特に優先度が高いのは以下です。

1. 設定・秘密情報の統制
2. トランザクション境界の是正
3. 機密情報を含むログの統制
4. レート制限、監査ログ、ヘルスチェックなど運用系クロスカット機能の実装統一
5. CI とテスト資産の信頼性回復

## 4. 点検結果

### 4.1 設定・秘密情報の統制

- 重要度: `Critical`
- 元レポート: `一部あり`
- 主な変更対象:
  - `libkoiki/core/config.py`
  - `app/core/sso_config.py`
  - `app/core/saml_config.py`
  - `.env.example`
  - `.env.production`
  - CI / デプロイ設定

#### 背景

Enterprise 品質では、アプリケーションコードそのものと同じくらい、設定と秘密情報の扱いが重要です。
設定値が「読み込める」だけでは不十分で、環境ごとに安全な値しか起動できないこと、
本番用秘密情報がソース管理や開発用テンプレートに混ざらないことが必要です。

#### 問題点

- `JWT_SECRET` に危険な既定値があり、本番でも起動拒否されません。
- `SSO_STATE_SIGNING_KEY`、`SAML_RELAY_STATE_SIGNING_KEY` も、強度やテンプレート値の検証がありません。
- `.env.production` がリポジトリに含まれており、運用設定とサンプル設定の境界が曖昧です。
- `SAML_SKIP_SSL_VERIFY=true` のような本番不適切設定が混入可能です。
- 設定の検証が存在確認中心で、安全性や環境整合性の検証まで到達していません。

#### Enterprise 品質上の懸念

- 本番で脆弱な設定のまま起動するリスクがあります。
- 設定値の誤配備や秘密情報漏えいに対して、予防的な防御が弱いです。
- 運用移管時に、設定の正解がコード・サンプル・実運用値に分散しやすくなります。

#### 目的

設定を「読める」状態から、「安全でない設定では起動できない」状態へ引き上げること。

#### 完了条件

- `production` 環境で既定シークレットやテンプレート値を検出した場合、起動失敗する。
- `*_SKIP_SSL_VERIFY=true` など、開発専用設定が本番で拒否される。
- 本番用設定ファイルをリポジトリ追跡対象から外し、Secrets 管理基盤へ移行している。
- `.env.example` は開発サンプルとしてのみ機能し、運用値を含まない。

#### テスト観点

- 本番環境変数に危険な既定値を与えたときに起動失敗すること。
- 開発環境では最小構成で起動できること。
- SSO/SAML の署名鍵が未設定、弱い、テンプレート値の各ケースで期待どおり検出されること。

### 4.2 トランザクション整合性の回復

- 重要度: `Critical`
- 元レポート: `あり`
- 主な変更対象:
  - `libkoiki/core/transaction.py`
  - `libkoiki/services/auth_service.py`
  - `libkoiki/services/password_reset_service.py`
  - `libkoiki/repositories/refresh_token_repository.py`
  - `libkoiki/repositories/password_reset_repository.py`
  - 認証系 endpoint / service

#### 背景

Enterprise システムでは、複数の更新操作が「途中まで成功して残る」状態を避ける必要があります。
特に認証、トークン失効、パスワードリセット、SSO/SAML 連携は、
1つの処理失敗がセキュリティ事故や問い合わせ増加に直結します。

#### 問題点

- service 層で `@transactional` を使っている一方、repository 内で個別 `commit()` を実行しています。
- endpoint と service の双方で `@transactional` を使う箇所があり、責務が二重化しています。
- refresh token rotation、password reset token 使用処理などで、部分コミットが残る可能性があります。

#### Enterprise 品質上の懸念

- 「旧トークンだけ失効し、新トークンは発行されない」などの半端な状態が起こりえます。
- 障害時の再実行・補償が難しくなります。
- 保守担当者が処理境界を追いにくく、原因分析コストが高くなります。

#### 目的

トランザクション境界を明確化し、1つの業務処理が原子的に成功または失敗する状態にすること。

#### 完了条件

- repository 内に `commit()` が残っていない。
- トランザクション責務が service か request 境界のどちらかに統一されている。
- refresh / revoke / password reset / SSO / SAML の代表フローで、途中例外時に DB が一貫状態へ戻る。

#### テスト観点

- refresh token rotation 中に例外を発生させた場合、旧トークン状態と新トークン状態が一貫すること。
- password reset 中に途中失敗しても、トークン使用済み状態とパスワード更新が食い違わないこと。
- SSO/SAML login フローで部分更新が残らないこと。

### 4.3 ログ設計と機密情報保護

- 重要度: `Critical`
- 元レポート: `あり`
- 関連ポリシー:
  - [`docs/backend-audit/BACKEND_LOGGING_SECURITY_POLICY_ja.md`](C:/Users/kataoka/Desktop/KOIKI-FW/KOIKI-FW-VSCodeProj/koiki-pyfw/docs/backend-audit/BACKEND_LOGGING_SECURITY_POLICY_ja.md)
- 主な変更対象:
  - `libkoiki/core/logging.py`
  - `libkoiki/core/error_handlers.py`
  - `libkoiki/core/security_logger.py`
  - `libkoiki/repositories/base.py`
  - `libkoiki/api/v1/endpoints/auth_password.py`
  - `libkoiki/services/user_service.py`

#### 背景

Enterprise 環境では、ログは運用資産であると同時に、情報漏えいの経路にもなります。
認証系・ユーザー系・監査系ログについては、「出すべき情報」と「出してはいけない情報」の線引きが必要です。

#### 問題点

- パスワードリセットトークン平文がログに出力されます。
- token 断片、メールアドレス、IP アドレス、更新データが広く記録されます。
- repository 作成時に ORM の `__dict__` をそのまま出力しており、機密情報が混入し得ます。
- DB 例外詳細がログに残り、内部構造や入力値が露出する可能性があります。

#### Enterprise 品質上の懸念

- セキュリティ事故時に、ログ保管先そのものが漏えい源になります。
- 監査ログと通常ログの境界が曖昧で、アクセス権限設計が難しくなります。
- セキュリティレビューや個人情報保護の観点で説明が難しくなります。

#### 目的

ログを「観測可能性を確保しつつ、機密情報を保護する設計」に再構成すること。

#### 完了条件

- password、token、secret、authorization header、session 識別子がログに出ない。
- email / IP / user agent の扱いが用途別に定義され、必要に応じてマスキングまたはハッシュ化される。
- 例外ログは保守に必要な情報を残しつつ、機密値を露出しない。
- 監査ログと通常アプリログの設計が分離されている。

#### テスト観点

- パスワードリセット要求、ログイン失敗、ユーザー更新時に機密値がログへ出ないこと。
- 構造化ログの sanitizer がネストした辞書にも機能すること。
- 代表的例外で、ユーザー向けレスポンスと内部ログの粒度が適切に分離されていること。

### 4.4 レート制限実装の統一

- 重要度: `Critical`
- 元レポート: `なし`
- 主な変更対象:
  - `libkoiki/core/rate_limiter.py`
  - `app/main.py`
  - 認証 / 一覧 / health 系 endpoint

#### 背景

Enterprise システムでは、認証 API、一覧 API、health endpoint などでレート制限の考え方が異なります。
その実装が二重化・分散していると、設定変更や運用判断の整合性が失われます。

#### 問題点

- endpoint の decorator はモジュールローカルの limiter を使用しています。
- 起動時に別 limiter を `app.state` に生成しており、設定が一元化されていません。
- Redis 利用、strategy、default limit の切替が実際の decorator に反映されません。

#### Enterprise 品質上の懸念

- 環境によってレート制限挙動が変わる可能性があります。
- 運用チームが期待する設定と実動作がズレます。
- セキュリティ試験、負荷試験の結果が安定しません。

#### 目的

レート制限の設定と実行経路を一本化し、環境差異のない運用品質を確保すること。

#### 完了条件

- limiter がアプリ全体で1系統に統一されている。
- Redis 戦略やデフォルト値の変更が実際の endpoint 制御に反映される。
- 認証 API と一般 API のレート制限方針が明文化されている。

#### テスト観点

- 同一クライアントからの連続アクセスで、設定どおり制限が掛かること。
- Redis 有無で期待どおりの fallback が行われること。
- 認証 API と一覧 API で異なる制限が正しく適用されること。

### 4.5 監査ログと相関 ID の成立

- 重要度: `High`
- 元レポート: `あり`
- 主な変更対象:
  - `libkoiki/core/middleware.py`
  - `libkoiki/core/logging.py`
  - `app/main.py`
  - 認証依存処理

#### 背景

Enterprise 運用では、障害解析、内部統制、セキュリティインシデント対応のために、
「どのリクエストが」「誰によって」「何をしたか」を追えることが必要です。

#### 問題点

- `RequestContextLogMiddleware` が定義されているにもかかわらず、アプリに組み込まれていません。
- `AuditLogMiddleware` は `request.state.current_user` を参照しますが、設定される経路がありません。
- 監査ログの前提と、実際の middleware 配線が一致していません。

#### Enterprise 品質上の懸念

- 監査ログが不完全で、実務上の証跡として弱いです。
- リクエスト単位のトレースができず、障害対応に時間がかかります。
- 監査や内部統制で要求される粒度に届きません。

#### 目的

request ID と principal 情報を含む、追跡可能な監査・アクセスログ基盤を成立させること。

#### 完了条件

- 全リクエストに request ID が付与される。
- 認証済みリクエストでは principal 情報が監査ログへ反映される。
- 監査ログに最低限必要な項目が統一フォーマットで出る。

#### テスト観点

- `X-Request-ID` が未指定でも生成され、応答に返ること。
- 認証済み endpoint で user_id が監査ログに含まれること。
- 除外対象 endpoint が監査ログポリシーどおり動作すること。

### 4.6 監視・ヘルスチェックの運用品質向上

- 重要度: `High`
- 元レポート: `あり`
- 主な変更対象:
  - `libkoiki/core/monitoring.py`
  - `app/main.py`
  - `app/api/v1/endpoints/sso_auth.py`
  - `app/api/v1/endpoints/saml_auth.py`

#### 背景

Enterprise システムでは、サービスが「動いている」ことと「トラフィックを受けられる」ことを分けて観測する必要があります。
また、認証基盤や外部依存を持つ場合は、内部状態と外部依存の両方を見られる必要があります。

#### 問題点

- Prometheus 計装は実装済みでも無効化されています。
- `/health` は DB / Redis / 外部IdP 到達性を確認せず常に healthy を返します。
- SSO / SAML health endpoint が内部設定情報を過剰に露出する構造です。
- in-memory メトリクスと Prometheus 方針が混在しています。

#### Enterprise 品質上の懸念

- Orchestrator が誤った正常判定を行う可能性があります。
- 障害時の初動切り分けに必要な可観測性が不足します。
- 監視設計の一貫性がなく、メトリクスの意味が不安定です。

#### 目的

本番運用で利用可能な、正確で過不足のない監視・health モデルを整備すること。

#### 完了条件

- `liveness` と `readiness` が分離されている。
- readiness で DB、Redis、必要な外部認証依存が確認される。
- metrics が正式に有効化され、業務メトリクスと技術メトリクスが整理されている。
- 公開 health endpoint が内部機密情報を過剰に返さない。

#### テスト観点

- DB 未接続、Redis 未接続、IdP 未到達などのケースで readiness が degraded / unhealthy になること。
- metrics endpoint が期待どおり公開されること。
- 外部向け health のレスポンスに機密設定が含まれないこと。

### 4.7 CI と品質ゲートの再設計

- 重要度: `High`
- 元レポート: `なし`
- 主な変更対象:
  - `.github/workflows/ci.yml`
  - `pyproject.toml`
  - `tests/`

#### 背景

Enterprise 品質では、「テストが存在する」ことよりも、「重要な機能変更が自動品質ゲートで止まる」ことが重要です。

#### 問題点

- CI で `libkoiki` の unit / integration が実行されていません。
- 重要な認証、DB、SSO/SAML の変更が pull request 時点で十分に検知されません。
- 品質ゲートの設計が、実際のリスクの高い領域と一致していません。

#### Enterprise 品質上の懸念

- 重要回帰が mainline に流入しやすいです。
- リリース判断の根拠が弱くなります。
- コードレビューで見落とした不具合を自動検知できません。

#### 目的

重要機能を確実に守る CI 品質ゲートを設計し直すこと。

#### 完了条件

- `libkoiki unit`, `app unit`, `integration`, `security checks` が CI の責務として整理されている。
- 認証、refresh token、password reset、SSO/SAML、DB 周辺が CI 対象に含まれる。
- 失敗時に修正すべきレイヤが分かるジョブ構成になっている。

#### テスト観点

- 各ジョブが独立に失敗原因を識別できること。
- 認証系回帰を意図的に起こしたときに CI で検知できること。
- DB を伴う integration が再現性を持って実行できること。

### 4.8 テスト資産の棚卸しと再整備

- 重要度: `High`
- 元レポート: `なし`
- 主な変更対象:
  - `tests/integration/app/api/test_auth_api.py`
  - `tests/unit/libkoiki/services/test_auth_service.py`
  - `tests/unit/libkoiki/services/`
  - `tests/integration/services/`

#### 背景

テストは量より信頼性が重要です。現行実装と乖離したテストを残すと、品質保証どころか誤誘導の原因になります。

#### 問題点

- 登録テストが現行スキーマ契約と一致していません。
- auth service unit test が現行実装に存在しない API を前提にしています。
- 「通っているテスト」と「意味のあるテスト」が一致していません。

#### Enterprise 品質上の懸念

- テスト結果の信頼性が低くなります。
- リファクタリング時の判断材料として使えません。
- 開発者が「何を壊してよいか」を誤解しやすくなります。

#### 目的

テスト資産を現行コードに整合させ、信頼できる回帰防止網へ戻すこと。

#### 完了条件

- 現行実装と一致しない obsolete test が整理されている。
- 重要業務フローについて、unit と integration の役割分担が整理されている。
- 認証/SSO/SAML/トークンまわりで、現行仕様に沿った trusted test が整備されている。

#### テスト観点

- 正常系だけでなく、認証失敗、期限切れ、再利用、部分失敗などの異常系を押さえていること。
- テストデータや fixture がトランザクション設計と整合していること。

### 4.9 API の運用品質改善

- 重要度: `High`
- 元レポート: `あり`
- 主な変更対象:
  - `libkoiki/api/v1/endpoints/users.py`
  - `libkoiki/api/v1/endpoints/todos.py`
  - response schema 群

#### 背景

Enterprise 向け API は、単に動くだけでなく、大量データ、クライアント拡張、運用制御を前提に設計される必要があります。

#### 問題点

- 一覧APIに `limit` 上限がありません。
- ページングメタデータがなく、クライアントが次ページ判定や総件数表示を独自実装する必要があります。
- エラー設計やレスポンス envelope の統一も限定的です。

#### Enterprise 品質上の懸念

- 大量取得に弱く、性能劣化や DoS 耐性の面で不利です。
- クライアント実装が不統一になります。
- API 契約を将来拡張しにくくなります。

#### 目的

一覧系 API を運用可能な契約へ引き上げ、拡張性・保守性を高めること。

#### 完了条件

- 一覧APIに最大ページサイズが定義される。
- `total`、`has_more`、`next_cursor` または同等のメタデータ方針が統一される。
- エラー応答と通常応答の構造が共通化される。

#### テスト観点

- 上限超過時の入力が拒否または補正されること。
- データ件数が多い場合でもページングが正しく機能すること。
- クライアントから見た契約が一貫していること。

### 4.10 DB スキーマとモデル統制

- 重要度: `Medium`
- 元レポート: `あり`
- 主な変更対象:
  - `alembic/versions/`
  - `libkoiki/models/user.py`
  - `libkoiki/models/associations.py`
  - `libkoiki/models/role.py`
  - `libkoiki/models/permission.py`

#### 背景

Enterprise システムでは、アプリケーションロジックだけで整合性を守るのではなく、
DB 制約とモデル定義でも不正状態を防ぐ必要があります。

#### 問題点

- `users.email` と `hashed_password` が nullable です。
- `user_roles` などの association table が重複定義されています。
- SQLAlchemy 2 系 / 旧記法が混在しています。
- timezone 付き時刻の一貫性が完全ではありません。

#### Enterprise 品質上の懸念

- 不正データが DB に入りやすくなります。
- マイグレーション保守が複雑化します。
- ORM アップグレードやモデル変更時に不具合を誘発しやすいです。

#### 目的

DB を業務整合性の最後の砦として機能させ、モデル定義の一貫性を高めること。

#### 完了条件

- 必須項目の nullable 制約が見直されている。
- association table の定義が一箇所に統一されている。
- ORM スタイル混在に対して統一方針が定義され、段階移行できる状態になる。

#### テスト観点

- マイグレーション適用が新規DB・既存DBの双方で成立すること。
- 一意制約、FK 制約、nullable 制約が想定どおり動作すること。

---

### 4.11 セキュリティ設定と実装の統合

- 重要度: `Medium`
- 元レポート: `あり`
- 主な変更対象:
  - `libkoiki/core/security_config.py`
  - `libkoiki/core/security.py`
  - `libkoiki/services/user_service.py`
  - `libkoiki/services/login_security_service.py`

#### 背景

セキュリティ設定が存在していても、実装がそれを参照していなければ、運用ポリシーとして機能しません。

#### 問題点

- password policy 設定があるのに、実際の判定はハードコードです。
- 設定を変えても挙動が変わらない箇所があります。
- セキュリティ方針がコード複数箇所に分散しています。

#### Enterprise 品質上の懸念

- セキュリティ要件変更に弱いです。
- 設定レビューと実装レビューの内容が一致しません。
- 監査時に「設定どおりに動いているか」を示しにくくなります。

#### 目的

セキュリティ方針を設定に集約し、実装をその設定に追従させること。

#### 完了条件

- password policy、login security policy が設定経由で制御される。
- 実装されていない設定項目が整理されている。
- セキュリティ関連設定と実装の対応表が作れる状態になっている。

#### テスト観点

- policy 値変更に応じてパスワード検証や lockout 挙動が変化すること。
- 設定変更が API / service / validation に一貫して反映されること。

### 4.12 障害時保守容易性の向上

- 重要度: `Medium`
- 元レポート: `一部あり`
- 主な変更対象:
  - `libkoiki/db/session.py`
  - `libkoiki/core/logging.py`
  - `libkoiki/core/security_metrics.py`
  - バックグラウンドタスク / 接続管理

#### 背景

Enterprise 品質では、障害を完全に防ぐことよりも、障害発生時に安全に劣化し、
短時間で切り分け・復旧できる構造であることが重要です。

#### 問題点

- DB 接続確認が単発で、起動再試行や運用上の復旧性が弱いです。
- 監査ログのローテーションや保持方針が明示されていません。
- セキュリティメトリクスなど一部状態がプロセス内メモリに閉じています。

#### Enterprise 品質上の懸念

- 一時的な依存障害でサービス起動が不安定になります。
- 再起動やスケールアウト時に運用情報が失われます。
- ログ・メトリクス・接続状態の扱いが運用設計へ十分落ちていません。

#### 目的

障害時の復旧容易性と保守容易性を高め、運用チームが扱いやすいシステムにすること。

#### 完了条件

- DB 接続の再試行 / pre-ping / 起動時ポリシーが整理されている。
- 監査ログの出力先、ローテーション、保持方針が確立している。
- プロセス内メモリに閉じた状態が必要に応じて Redis / DB / metrics backend へ移されている。

#### テスト観点

- 起動時に一時的に DB が未接続でも、方針どおり再試行または明確に失敗すること。
- マルチワーカーや再起動時に重要な運用状態が失われないこと。
