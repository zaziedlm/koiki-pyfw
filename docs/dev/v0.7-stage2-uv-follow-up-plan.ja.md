# KOIKI-FW v0.7 Stage 2 `uv` follow-up plan

## 目的

Stage 2 の `uv` 実移行により、root workspace、`uv.lock`、local docs、CI の標準経路は `uv` に切り替わった。

一方で、Docker、security audit runbook、古い helper、component package metadata、履歴文書には Poetry 前提の記述や設定が残っている。

この文書は、残存 Poetry 要素を次工程で段階的に整理するための follow-up 計画である。

## 現在の基準

現行の標準経路:

- 依存同期: `uv sync`
- lockfile 固定同期: `uv sync --locked`
- テスト実行: `uv run pytest ...`
- CI 実行: `uv sync --locked` と `uv run --locked ...`
- lockfile 正本: `uv.lock`

現行の非標準経路:

- `poetry.lock`
- `poetry install`
- `poetry run ...`
- `poetry build`
- `[tool.poetry]`
- `poetry-core`

## 分類ルール

Poetry 残存は次の3種類に分類する。

### A. 実運用残存

現在も人、CI、Docker、script が実行し得る Poetry 前提。

原則として follow-up で更新または削除する。

例:

- Dockerfile
- security audit command
- test guide
- SAML setup guide
- helper script
- package metadata

### B. component metadata 残存

component package を build / install するための `pyproject.toml` に残る Poetry 固有設定。

`pyproject.toml` 自体は削除しない。
Poetry 固有部分だけを PEP 517 / PEP 621 の別 backend へ置き換える。

### C. 履歴文書

過去の設計、過去の release note、移行経緯を説明する文書。

原則として内容を書き換えない。
必要な場合だけ、冒頭に historical reference であることと、現行手順へのリンクを追記する。

## Follow-up タスク

### FU-1: Docker / Compose の `uv` 化

優先度:

- 最優先

理由:

- `poetry.lock` は削除済みであり、Dockerfile が `COPY poetry.lock` や `poetry install` を参照している場合、Docker build が破綻する。

対象:

- `Dockerfile`
- `Dockerfile.unified`
- `docker-compose*.yml`
- `scripts/verify-dockerfile-unified.sh`
- `scripts/verify-dockerfile-unified.ps1`

作業:

- Poetry install step を削除する
- `uv` install step を追加する
- `COPY poetry.lock` を `COPY uv.lock` に変更する
- `poetry install` を `uv sync --locked` に変更する
- `POETRY_*` env / build arg を削除または `UV_*` に置換する
- Docker 起動 import path を確認する
  - 互換導線: `app.main:app`
  - 新標準導線: `koiki_ref_app.asgi:app`

検証:

- `docker build`
- `docker compose build app`
- 可能なら `docker compose up` による起動確認

完了条件:

- Docker build が `poetry.lock` なしで成立する
- Dockerfile 内の install / lockfile 参照が `uv` 前提になっている

実施結果:

- `Dockerfile` と `Dockerfile.unified` を Poetry から `uv` ベースへ更新した
  - `COPY poetry.lock` を `COPY uv.lock` に変更
  - Poetry install step を削除
  - `uv` binary を `ghcr.io/astral-sh/uv:0.11.7` からコピー
  - `uv sync --locked` で `/app/.venv` を作成
  - runtime `PATH` に `/app/.venv/bin` を追加
- Docker / Compose の backend 起動 import path を `koiki_ref_app.asgi:app` に更新した
- `scripts/verify-dockerfile-unified.sh` と `scripts/verify-dockerfile-unified.ps1` から `POETRY_MAX_WORKERS` build arg を削除した
- BuildKit cache mount 中の `uv cache clean` が `Resource busy` で失敗したため、Dockerfile 内から削除した

検証結果:

- `docker build --file Dockerfile.unified --target dev --tag koiki-pyfw-app:dev-uv-check .`
  - 成功
- `docker build --file Dockerfile.unified --target production --build-arg INSTALL_MAIN_ONLY=true --tag koiki-pyfw-app:prod-uv-check .`
  - 成功
- `docker build --file Dockerfile --target production --tag koiki-pyfw-app:prod-dockerfile-uv-check .`
  - 成功
- `docker run --rm --entrypoint python koiki-pyfw-app:prod-uv-check -c "import libkoiki, koiki_ref_app; print(libkoiki.__name__, koiki_ref_app.__name__)"`
  - 成功
- `docker run --rm --entrypoint sh koiki-pyfw-app:prod-uv-check -c "which python && which uvicorn && which alembic"`
  - 成功
  - `/app/.venv/bin/python`
  - `/app/.venv/bin/uvicorn`
  - `/app/.venv/bin/alembic`
- `docker compose -f docker-compose.yml build app`
  - 成功
- `docker compose -f docker-compose.unified.yml --profile dev build app-dev`
  - 成功

状態:

- `完了`

### FU-2: `components/libkoiki` package metadata 整理

優先度:

- 高

対象:

- `components/libkoiki/pyproject.toml`

作業:

- `[tool.poetry]` を削除する
- `[tool.poetry.group.*]` を削除する
- `[tool.poetry.dependencies]` を削除する
- `poetry-core` build backend を削除する
- `setuptools` または別の PEP 517 backend に置き換える
- `src` layout の package discovery を明示する

注意:

- `components/libkoiki/pyproject.toml` 自体は削除しない
- workspace member として `libkoiki` を install / build するために必要である

検証:

- `uv lock --check`
- `uv sync --locked --group dev --group test`
- `uv run python -c "import libkoiki"`

完了条件:

- `components/libkoiki/pyproject.toml` から Poetry 固有設定が消えている
- `libkoiki` が workspace package として import できる

実施結果:

- `components/libkoiki/pyproject.toml` から Poetry 固有設定を削除した
  - `[tool.poetry]`
  - `[tool.poetry.group.dev]`
  - `[tool.poetry.group.test]`
  - `[tool.poetry.dependencies]`
  - `poetry-core` build backend
- build backend を `setuptools.build_meta` に変更した
- `src` layout の package discovery を `[tool.setuptools.packages.find]` で明示した
- component の `[project]` metadata と optional `dev` dependencies は維持した

検証結果:

- `uv lock --check`
  - 成功
- `uv sync --locked --group dev --group test`
  - 成功
  - `libkoiki` が `setuptools.build_meta` で build / reinstall された
- `uv run --locked python -c "import libkoiki; print(libkoiki.__name__)"`
  - 成功
- `DEBUG=true uv run --locked pytest components/libkoiki/tests --collect-only`
  - 成功
  - 115 tests collected

補足:

- `DEBUG` 未上書きの場合、Codex 実行プロセス環境に継承されていた `DEBUG=release` により `Settings.DEBUG` の bool parse error が発生する。
- repository の `.env` には `DEBUG=release` は存在しない。
- これは FU-2 の package metadata 変更とは別件の実行環境課題であり、Codex 側の検証では `DEBUG=true` を明示して実行する。

状態:

- `完了`

### FU-3: Security audit 導線の `uv` 化

優先度:

- 高

対象:

- `SECURITY_AUDIT_COMMANDS.md`
- 必要に応じて security audit 用 workflow

作業:

- `poetry install --with security` を `uv sync --group security` に置換する
- `poetry run pip-audit` を `uv run pip-audit` に置換する
- `poetry run bandit` を `uv run bandit` に置換する
- `poetry show --outdated` / `poetry show --tree` を `uv` の診断コマンドに置換する
  - 例: `uv tree`
  - 必要に応じて `uv pip list`
- `safety` の扱いを決める
  - 現在の標準 security group には `safety` を含めない
  - 原則は `pip-audit` と `bandit` を標準とする
  - `safety` を継続採用する場合は dependency group へ明示追加する
- セキュリティ修正時の dependency update 手順を `uv add` / `uv lock` / `uv sync` 前提に更新する

検証:

- `uv sync --group security`
- `uv run pip-audit`
- `uv run bandit -r .`

完了条件:

- `SECURITY_AUDIT_COMMANDS.md` の実行可能コマンドが `uv` 前提になっている
- security group の実依存と文書が一致している
