# KOIKI-FW v0.7 ディレクトリ再編計画

## 目的

本書は、`v0.7` 系で実施する具体的なリポジトリ再編計画を定義するものです。

主目的は、KOIKI-FW を他プロジェクトへ次の 2 つの形で提供しやすくすることです。

1. 再利用可能なフレームワークライブラリとして提供する
2. そのフレームワークを合成して使う参照アプリケーションとして提供する

現状のリポジトリでも `libkoiki/` と `app/` は分かれていますが、パッケージング、起動、マイグレーション、テストの責務はまだ部分的に混在しています。本計画では、その配布境界を明示化します。

## 文書同期ルール

本日本語版と英語版は、同期対象の計画書ペアとして扱います。

- `v0.7-directory-reorganization-plan.ja.md`

- ブランチ前提と基準ブランチ名
- ロードマップの Stage と Deliverable
- 命名方針とレイアウト方針

## 目標状態

再編完了時には、次の状態を目指します。

- `libkoiki` が、公開可能なクリーンなフレームワークパッケージになっている
- 現在の `app/` が、名前付きの参照アプリケーションパッケージになっている
- 案件固有アプリケーションの置き場が、トップレベルの `apps/` として分離されている
- `frontend/` は、バックエンドテンプレートと組になるスターターフロントエンドとしてリポジトリ直下に残る
- リポジトリ直下は、開発、CI、ドキュメント、ローカル運用のための workspace になる
- 実行時の起動は、`app.main` にハードコードされた構成ではなく app factory ベースになる
- component 単位でマイグレーション、テスト、ドキュメントの所在が理解しやすくなる

## 現在の構造上の問題

現状のリポジトリには、実務上次の問題があります。

- ルートの `pyproject.toml` は workspace 的なファイルになっているが、公開可能なパッケージ定義としては整理されていない
- `app/` は汎用名であり、しかも現時点では通常の Python package にもなっていない
- `libkoiki` には、フレームワーク関心と具体的なサンプル/参照 API の両方が含まれている
- 起動の合成が `app/main.py` に集中しており、外部プロジェクトでの再利用性を下げている
- ORM の拡張ポイントが import 時の副作用に依存している
- テストは存在するが、責務ごとの所有境界がディレクトリ上で常に明確とは言えない

## 目標構成における責務ルール

### `components/libkoiki`

この component は、次を責務として持ちます。

- 再利用可能なフレームワーク挙動
- 共通の認証、ロギング、ミドルウェア、レート制限、監視
- 再利用可能な DB / session 抽象
- 再利用可能な schema、repository、service、および拡張契約
- 他プロジェクトが利用できる app factory のプリミティブと合成フック

この component に持たせるべきでないものは次です。

- 案件固有の SSO provider
- 業務固有 API endpoint
- フレームワーク契約の本質ではないサンプルドメイン CRUD
- 参照アプリ固有の起動詳細

### `components/koiki_ref_app`

この component は、次を責務として持ちます。

- 現在のアプリケーション固有 API 挙動
- 現在の SSO / SAML 統合
- 参照アプリケーションの起動
- 案件寄りの model と migration
- 参照アプリケーションの設定デフォルト

この component は、`libkoiki` の上にどうアプリを組み立てるかを示す正規の参照実装です。

### `apps/`

このトップレベル領域は、次を責務として持ちます。

- 案件固有アプリケーション
- 顧客固有またはデプロイ固有のバックエンド実装
- 上流の再利用資産としては自動的に扱うべきでない downstream 側のアプリケーションコード

この領域は、再利用可能なフレームワークやテンプレートコードのデフォルト配置先にしてはいけません。

### リポジトリ直下

ルートは、次を責務として持ちます。

- workspace 用ツール設定
- CI
- Docker とローカル運用
- 組みになった `frontend/` スターター実装
- 複数 component と複数 application を跨ぐ E2E テスト
- リポジトリ全体のドキュメント

リポジトリ直下は、配布対象そのもののメインランタイムパッケージであるかのように振る舞ってはいけません。

## 現行構成からの具体的な対応表

### フレームワーク component

現状:

- `libkoiki/`

移行先:

- `components/libkoiki/src/libkoiki/`

既存のフレームワークコードの大半は維持しつつ、`src/` ベースのパッケージ構成へ移します。

### 参照アプリケーション component

現状:

- `app/`
- `app/main.py`
- `app/core/`
- `app/api/v1/`
- `app/models/`
- `app/repositories/`
- `app/services/`
- `app/schemas/`

移行先:

- `components/koiki_ref_app/src/koiki_ref_app/`

これにより、曖昧な名前である `app` をやめ、外部プロジェクトでも安定して import 可能なパッケージ名にします。

### マイグレーション

現状:

- ルート `alembic/`
- ルート `alembic.ini`

移行先:

- `components/koiki_ref_app/alembic/`
- `components/koiki_ref_app/alembic.ini`

理由:

- 現在の migration は、抽象的な framework スキーマではなく、合成済みアプリケーションスキーマに結び付いているため

### エントリポイント

現状:

- ルート `main.py`
- `app/main.py`

移行先:

- `components/koiki_ref_app/src/koiki_ref_app/asgi.py`
- `components/koiki_ref_app/src/koiki_ref_app/app_factory.py`

移行期間中は、必要であればルート側に互換ラッパーを残してもよいですが、それを正規の起動経路として扱うべきではありません。

### テスト

現状:

- `tests/unit/app/`
- `tests/unit/libkoiki/`
- `tests/integration/app/`
- `tests/integration/services/`
- `tests/e2e/`

移行先:

- `components/libkoiki/tests/unit/`
- `components/libkoiki/tests/integration/`
- `components/koiki_ref_app/tests/unit/`
- `components/koiki_ref_app/tests/integration/`
- ルート `tests/e2e/`

理由:

- 所有境界が明確になる
- component 単位で CI を小さく回しやすくなる
- 再利用フレームワークのテストが、アプリのレイアウト前提に引きずられにくくなる

### Frontend

現状:

- `frontend/`

移行先:

- `frontend/` をリポジトリ直下に維持する

理由:

- frontend は Python component 群とはツールチェーンもランタイムも異なる
- ルートに置くことで、人間にも AI エージェントにも見通しがよい
- frontend はバックエンドテンプレートと組みではあるが、`libkoiki` パッケージの一部ではない

### 案件固有アプリケーション

現状:

- 案件固有アプリ用の専用トップレベル領域はまだ存在しない

移行先:

- `apps/`

理由:

- `components/` は再利用可能な framework、reference、platform component に集中させるべき
- `apps/` は案件固有アプリ実装の明確な置き場になる
- これにより、上流の再利用資産と downstream の案件コードの境界が、人間にも AI エージェントにも見えやすくなる

## 必要な内部リファクタリング

単にディレクトリを移すだけでは不十分です。以下のコードレベル再編も必要です。

### 1. 明示的な app factory 合成の導入

framework 側には、次のような合成プリミティブを追加します。

- `libkoiki.app_factory.create_base_app(...)`
- `libkoiki.app_factory.install_framework_middlewares(...)`
- `libkoiki.app_factory.include_framework_routers(...)`

その上で参照アプリ側では、次の順に app を組み立てます。

1. `libkoiki` から base app を構築する
2. 参照アプリ固有の設定と統合を登録する
3. 参照アプリ側の router を取り込む
4. 参照アプリ側の background task を組み込む

これは、1 つの具体的な `app.main` が全責務を持つ現状より望ましい形です。

### 2. ORM の副作用的バインディングを明示 bootstrap に置き換える

将来的に避けるべき現行パターン:

- `app.models` を import すると `libkoiki.models.UserModel` が書き換わる

推奨ターゲット:

- `koiki_ref_app.bootstrap.orm` モジュールを追加する
- `register_model_extensions()` のような明示関数を定義する
- app 起動時と migration 起動時に、その bootstrap を呼ぶ

狙いは、暗黙の import 順依存をなくすことです。

### 3. framework API と sample/reference API を分離する

`libkoiki` 内では、次の整理を推奨します。

- auth、user、permission、framework monitoring は、本当に framework 契約なら残す
- `todos` などの sample-domain endpoint は次のどちらかへ移す
  - `components/koiki_ref_app`
  - `examples/`

framework を説明するためだけの API は、デフォルトの framework surface に含めるべきではありません。

### 4. パッケージメタデータを標準化する

`libkoiki` について:

- `pyproject.toml` を正とする
- レガシーな `setup.py` は削除する
- 依存定義を 1 か所に統一する

参照アプリについて:

- 公開可能な実パッケージ定義を持たせる
- テスト専用依存を runtime dependency から外す
- `libkoiki` を通常依存として宣言する

ルート workspace について:

- 軽量な workspace 用 `pyproject.toml` を維持する
- それ自体を配布対象のアプリ本体として見せない

## 推奨実行順序

大きなディレクトリ移動から始めるべきではありません。

このリポジトリでは、次の順序が最も安全です。

1. `v0.6` 保守線と `v0.7` 開発線のブランチ運用を安定化する
2. パッケージメタデータと依存責務を整理する
3. Poetry から `uv` へ依存管理を移行する
4. 大きなディレクトリ移動の前に app factory 合成を導入する
5. テンプレート配布モデルを明示的に定義する
6. `components/` と `src/` に移し、案件固有アプリは `apps/` に分ける
7. migration、scripts、CI を新構成へ合わせる
8. 暫定互換 shim を除去する

この順序が安全な理由は次の通りです。

- ブランチ混乱を、構造変更前に解消できる
- `uv` 移行を、component 移動によるパス変更が増える前に済ませられる
- import 書き換え前に app factory 抽出を完了できる
- テンプレート方針を、パッケージングと docs の固定前に決められる

使い分け:

- `feature/*` は構造変更や機能面の大きめな作業用

### マージポリシー

- 侵襲的な `v0.7` 作業を始める前に、`v0.6.x` 安定タグを打つ

## 本計画における `uv` 移行の位置づけ

`uv` 移行は早い段階で行うべきですが、最初の一手ではありません。

推奨ルール:

- まずパッケージ責務を整理し、メタデータの曖昧さを解消する
- 次に依存管理を `uv` に移す
- その後で component を最終的な `components/` 配置へ移す

この順序が望ましい理由は、現状まだ次が混在しているためです。

- ルートの workspace 的 `pyproject.toml`
- 別立ての `libkoiki/pyproject.toml`
- レガシーな `libkoiki/setup.py`
- runtime / test dependency が曖昧な `app/pyproject.toml`

依存管理を正規化する前に multi-component workspace に移すと、移行コストが不必要に上がります。

### `uv` 移行のゴール

`v0.7` において、`uv` 移行の到達点は次です。

- `uv.lock` が正規 lockfile になる
- `uv sync` と `uv run` が標準のローカル開発コマンドになる
- ルートの workspace 設定が、最終的な multi-component 構造に揃う
- component ごとの依存定義は `pyproject.toml` に保持する
- Poetry 依存の CI / ローカル手順を除去する

### 大きな移動前に行う `uv` 移行範囲

`components/` へ移す前に、次を行います。

- `libkoiki/setup.py` を削除する
- ルート `pyproject.toml` を workspace 意図に合わせて簡素化する
- `libkoiki/pyproject.toml` を正規化する
- 将来の参照アプリパッケージメタデータを正規化する
- Poetry 前提の CI / 開発コマンドを `uv` に置き換える

## 本計画におけるテンプレート戦略

テンプレート設計は、最終的な component 移動が完了する前に確定させるべきです。

推奨する提供モデルは次です。

- `libkoiki`
  - 再利用可能な framework package
- `koiki_ref_app`
  - 新規プロジェクト用の参照実装兼スターターテンプレート
- `frontend/`
  - `koiki_ref_app` と組になる root-level のスターターフロントエンド

つまり downstream 側の利用は、単に次だけではありません。

- `libkoiki` を install する

それに加えて、次を含みます。

- ローカル可視性と AI エージェント親和性が必要な場合、`koiki_ref_app` から始める
- 標準 UI / BFF スターターが必要な場合、`frontend/` も一緒に採用する

### テンプレートの目標

テンプレートパッケージは、次を保持すべきです。

- framework 合成構造がローカルで見通せること
- Agent guidance と task routing docs が同梱されていること
- 起動、router 合成、拡張ポイントの具体例があること
- starter 用の migration、test、運用 script が適切に含まれていること

テンプレート全体としては、次を保持すべきです。

- `koiki_ref_app` をバックエンドスターターとして持つこと
- `frontend/` を root-level のフロントエンドスターターとして持つこと
- AI コーディングエージェントが両レイヤーで効果的に作業できるための guidance を持つこと

一方でテンプレートは、次になってはいけません。

- 関係のない過去案件の遺物の寄せ集め
- `libkoiki` に属すべき framework-only ロジックの新しい置き場

### テンプレート更新モデル

`v0.7` 時点では、テンプレートの利用は copy-first です。

つまり:

- downstream プロジェクトは、テンプレート構造を複製/採用して開始することを前提とする
- downstream 側は、upstream と live に同一 workspace を共有することを必須としない
- upstream 側の改善は、後で手動適用できるようドキュメント化・パッケージ化しておく

本計画では、upstream と downstream の間で live に MCP ベース同期することは要求しません。

## Agent Guidance 戦略

このリポジトリは、AI エージェント向け guidance を次の 3 系統で支えられるように再編します。

### Skill クラス

- maintainer
  - `libkoiki` と upstream テンプレート自体を変更する人向け
- template
  - `koiki_ref_app` と組になった `frontend/` スターターを触る人向け
- consumer
  - テンプレート採用先または `libkoiki` 利用先の downstream プロジェクト向け

### 段階的な導入方針

Skill の分割は、一度に完成させる必要はありません。

推奨する段階導入:

1. 構造移行中も、現在の共有 guidance をそのまま動かす
2. まず maintainer と template を分ける
3. 新しいテンプレート境界が安定してから consumer 向け guidance を入れる

この段階導入は、効果を見ながら進め、初期の過剰設計を避けるために意図したものです。

### Skill 再編フェーズ

Skill の再編は、次の 3 フェーズで進めます。

#### Phase A: 現行 Skill を維持しつつ役割を明確化する

このフェーズでは:

- 現行 Skill をそのまま使える状態で維持する
- すぐに rename を強制しない
- 各 Skill に将来の役割対応を注記する

この時点での読み替えは次です。

- `koiki-project-overview`
  - 一時的な cross-repository overview / routing skill
- `koiki-libkoiki-feature-work`
  - maintainer 側の framework skill の初期形
- `koiki-app-feature-work`
  - template-backend skill の初期形
- `koiki-auth-security`
  - 将来フェーズでも残る横断 skill
- `koiki-testing`
  - 将来フェーズでも残る横断 skill

Phase A の目的は、分類の完成ではなく継続性の確保です。

#### Phase B: maintainer と template を分離する

ここが最初の大きな再編です。

追加・抽出する対象:

- maintainer overview
- maintainer framework work
- maintainer template-backend work
- maintainer template-frontend work
- template overview
- template backend feature work
- template frontend feature work

このフェーズで初めて、root-level の `frontend/` スターターを Skill モデルに明示的に組み込みます。

その理由は、template work が backend-only ではないためです。

- `koiki_ref_app` を理解する必要がある
- 組になった `frontend/` を理解する必要がある
- 変更を `libkoiki` に上げるべきか、template 内に留めるべきかの判断 guidance が必要になる

#### Phase C: consumer 向け Skill を追加する

このフェーズは、新しい component 境界と template 境界が安定した後に行います。

Consumer skill の主題は次です。

- downstream プロジェクトが `libkoiki` をどう採用するか
- downstream プロジェクトが template をどう採用/複製するか
- downstream 側の変更をローカルに留めるべきか、upstream 提案すべきかをどう判断するか
- upgrade と互換性の考え方をどう理解するか

Consumer skill は、動いている最中の upstream 構造ではなく、安定した upstream 構造を前提にすべきため、意図的に後ろへ回します。

### 移行期間中の互換方針

移行中は、次を守ります。

- legacy skill を早く消しすぎない
- 新 skill 導入中も旧エントリポイントを維持する
- 推奨置き換え先がある場合は、旧 skill を transitional と明記する
- 旧 skill 名から新カテゴリへの対応表を文書化する

この方針は、人間にとっても AI エージェントにとっても、急な skill 置換による混乱を避けるために重要です。

### Skill family の最終目標構成

長期的な目標構成は次です。

- maintainer
  - overview
  - libkoiki
  - template-backend
  - template-frontend
  - auth-security
  - testing
- template
  - overview
  - backend-feature
  - frontend-feature
  - auth-integration
  - testing
- consumer
  - overview
  - libkoiki-integration
  - auth-setup
  - upgrade-guide

この最終形を、最初から全部作る必要はありません。

優先順位は次です。

1. まず現在の共有 skill セットを維持する
2. maintainer と template の区別を確立する
3. template モデルが安定した後で consumer guidance を追加する

## 本計画における MCP の位置づけ

MCP 対応は、今回の `v0.7` 再編スコープからは明示的に外します。

理由:

- まずリポジトリの構造とパッケージングを明確にする必要がある
- Skill とローカル guidance だけでも、当面の実務価値は十分出せる
- MCP は、template と guidance の構造が安定してから再検討すべき

したがって:

- MCP 設計は `v0.7` を止める要件にしない
- 現在の migration phase を MCP rollout と結び付けない
- repository structure、template model、skills model が固まった後の follow-up initiative として扱う

## 移行フェーズ

### Phase 0: 目標形の確定

ファイル移動の前に、次を確定します。

- このディレクトリ構成を正式採用する
- 参照アプリの最終パッケージ名を決める
- どの API を `libkoiki` に残すか決める
- `todos` を framework、example、reference-app のどこに置くか決める

### Phase 1: 大移動前のパッケージ定義整理

先に行うこと:

- `libkoiki/setup.py` を削除する
- `app/` を実パッケージにする、または新しい target package へ置き換える
- `app/pyproject.toml` の runtime / dev dependency を整理する
- ルート workspace の意図を明示する

このフェーズは、大きなパス移動前の曖昧さ除去が目的です。

### Phase 2: app factory 導入

次に行うこと:

- framework 側 app factory helper を作る
- 参照アプリ側 `create_app()` を作る
- 現在の `app/main.py` の起動ロジックを明示的な bootstrap module へ移す
- 既存コマンド互換のために wrapper を一時維持する

このフェーズにより、大きな path 移動前に構造依存を軽くします。

### Phase 3: `components/` + `src/` へ移動し、`apps/` を導入する

app factory 導入後に、次を行います。

- `libkoiki/` を `components/libkoiki/src/libkoiki/` へ移す
- `app/` を `components/koiki_ref_app/src/koiki_ref_app/` へ移す
- `apps/` を案件固有アプリの予約領域として導入する
- component 単位テストを各 component 配下へ移す
- import、tooling、Docker path を更新する

### Phase 4: migration と script の移設

続いて、次を行います。

- Alembic を参照アプリ component 配下へ移す
- migration command と CI を更新する
- docker entrypoint と local script を更新する

### Phase 5: 暫定互換の除去

最後に、次を行います。

- 不要になったルート runtime wrapper を削除する
- import shim を削除する
- 残っている side-effect model bootstrap を削除する

## 推奨命名方針

### フレームワーク package

維持:

- `libkoiki`

理由:

- すでに定着している
- 再利用 library の責務が分かりやすい

### 参照アプリケーション package

推奨名:

- `koiki_ref_app`
- `koiki_sso_reference_app`

避けるべき名前:

- `app`
- `backend`
- `sample`

理由:

- 外部利用者にとって、安定して曖昧さのない import 名が必要なため

## CI とツール調整

構成変更後は、次の方針にします。

- CI は `components/libkoiki` と `components/koiki_ref_app` を別々に検証する
- ルート E2E は、component-level check が通った後に実行する
- coverage は component ごとに出し、必要なら集約する
- Docker build context はルートのままとし、startup module は参照アプリ component を向く

## 期待される効果

この再編により、次が得られます。

- 公開可能な framework package が整理される
- 参照アプリケーション package が明確になる
- 外部プロジェクトでの再利用性が上がる
- コントリビュータのオンボーディングが容易になる
- import 順依存の脆さが減る
- テストと migration の所有境界が明確になる

## `v0.7` の非目標

本計画では、次は要求しません。

- リポジトリ分割
- すべての package を直ちに PyPI 公開すること
- ドメインモデル全面再設計
- framework 内部の全面書き換え

狙いは、まず境界と配布形態を改善することです。

## 再編完了後の follow-up

Stage 0 から Stage 7 までの本線タスクにより、ディレクトリ再編そのものは完了した。
ただし、これはあくまで「構造再編トラックの完了」であり、運用安定化と実行検証は後続で回収する。

再編完了後に扱う後続テーマは次です。

- `uv` 実移行
  - 方針確定までは完了しているが、`uv.lock`、`dependency-groups`、`tool.uv.workspace`、CI 上の `uv` 実行までは未完了
- dependency 同期後の runtime 検証
  - `pytest`
  - Alembic 実行
  - `from app.main import app`
  - `from koiki_ref_app.asgi import app`
  - `import libkoiki`
  - `import koiki_ref_app`
- CI / GitHub Actions の実行確認
  - path 更新は完了しているが、workflow 実行結果の確認は未実施
- Docker / Compose の実起動確認
  - path と mount の追随は完了しているが、実コンテナ起動確認は未実施
- 互換導線の最終整理
  - `app/__init__.py`
  - `app/main.py`
  - root `main.py`
  - `app.main:app`
  をいつ正式廃止するかを、運用告知を含めて決める必要がある
- 入口ドキュメントと履歴資料の整理
  - root `AGENTS.md` は現状の `components/` 構造とずれている
  - 一部の docs / design note / SAML guide には旧 `app.*` 前提の記述が残っている

このため、再編後の第 2 トラックは「構造変更」ではなく、次の 3 つを中心に進める。

1. dependency / toolchain の実移行
2. runtime / CI / container の実行検証
3. 互換層と古い導線の正式整理
