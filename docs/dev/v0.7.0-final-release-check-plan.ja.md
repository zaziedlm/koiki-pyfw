# v0.7.0 release final check / tag preparation plan

作成日: 2026-05-05

対象ブランチ: `topic/v070-final-check-plan`

## 1. 目的

`DM-13` で v0.7.0 release preparation は完了したが、tag / GitHub release を切る前に、release candidate としての品質を多面的に確認する。

このタスクでは、リリース直前に実行する確認観点、必須チェック、任意チェック、証跡の残し方、tag 作成判断を明確にする。

## 2. スコープ

対象:

- clean checkout / clean working tree での validation
- version metadata / FastAPI metadata / `/health` response の整合確認
- release note / README / architecture docs の最終確認
- Todo API の最終配置判断
- unit / collect-only / integration candidate の実行判断
- container build / run / UI smoke
- security / dependency audit の残件確認
- tag / GitHub release 作成判断

非対象:

- v0.7.0 直前の大きな runtime behavior 変更
- `app.main:app` 互換終了
- Todo API の app layer 移動、optional router 化、generic base capability 化
- downstream `apps/` の具体実装追加

不具合が見つかった場合は、release final check の中で大きく直さず、修正 PR を分ける。

## 3. リリース前提判断

v0.7.0 では次を維持する。

- 標準 ASGI entrypoint は `koiki_ref_app.asgi:app`
- `app.main:app` は compatibility wrapper として維持
- root `app/` へ新規実装を追加しない
- `apps/` は downstream / customer-specific backend API の予約領域として維持
- Todo API は `components/libkoiki/` の framework sample / starter capability として明示維持

Todo API については、今回の最終チェックで「移動しないこと」を再確認する。これは、新規 business API を `libkoiki` に置く前例ではなく、v0.7.0 の sample / starter / smoke 対象として維持する判断である。

## 4. 必須チェック

### REL-01: checkout / branch hygiene

目的:

- release 判定対象が最新の `dev/v0.7` 系であることを確認する
- dirty working tree や意図しない差分を残した状態で release 判定しない

確認例:

```powershell
git status --short --branch
git log --oneline --decorate -5
git diff --check
```

完了条件:

- release 判定前の作業ツリーが clean
- tag 作成対象 commit が明確
- topic branch の計画差分は release 判定対象へ混入させる前に review 済み

### REL-02: version / runtime metadata

目的:

- package metadata、FastAPI metadata、runtime response が `0.7.0` で揃っていることを確認する

確認例:

```powershell
uv lock --check
uv run --locked python -c "import libkoiki; print(libkoiki.__version__)"
uv run --locked python -c "from koiki_ref_app.asgi import app; print(app.version)"
```

container 起動後:

```powershell
Invoke-RestMethod http://localhost:8000/health
Invoke-RestMethod http://localhost:8000/
```

完了条件:

- `libkoiki.__version__` が `0.7.0`
- FastAPI `app.version` が `0.7.0`
- `/health` と `/` の version 表示が `0.7.0`
- `uv.lock` と pyproject metadata に不整合がない

### REL-03: Todo API final placement

目的:

- Todo API を v0.7.0 では `libkoiki` の sample router / starter capability として維持する判断が、コード・docs・agent guidance の間で矛盾していないことを確認する

確認例:

```powershell
rg -n "todo|todos|Todo" components/libkoiki/src components/koiki_ref_app/src components/libkoiki/tests components/koiki_ref_app/tests docs/agent README.md docs/design_kkfw_0.7.0.md docs/releases/KOIKI-FW_0.7.0.md -S
rg --files components/libkoiki/src components/koiki_ref_app/src | rg "todo|router|dependencies|app_factory|asgi"
```

確認観点:

- Todo endpoint / model / schema / repository / service が現在の sample capability として説明できる
- `koiki_ref_app` は Todo router を reference app として利用している
- Todo を新規 business API の `libkoiki` 配置前例として説明していない
- Todo の app layer 移動や optional router 化を v0.7.0 release blocker にしていない
- frontend Todo smoke が release validation として使える

完了条件:

- v0.7.0 では Todo API を `libkoiki` sample router として明示維持する判断に矛盾がない
- コード移動なしで release 判定へ進める
- 後続で再判断する場合の候補が docs 上で分離されている

- agent guidance contract test が成功
- collect-only が成功
- 実行可能な unit / integration scope が成功
- 既知 warning と新規 failure を分けて記録している

### REL-07: security / dependency audit

目的:

- release 前に既知 security / dependency 残件と新規検出を分けて確認する

確認例:

```powershell

uv run --locked bandit -c pyproject.toml -r components apps tests
```

完了条件:

- 新規 high severity がない

- `pip 26.0.1 / CVE-2026-3219` など監査環境側の既知残件は、runtime dependency と分けて扱う
- JWT test secret length warning など既知 warning は、今回の release blocker かどうかを明確にする

## 5. 任意チェック

実行環境と時間が許す場合:

- frontend build
- browser e2e smoke の自動化
- DB integration full scope
- clean clone 相当の別 directory validation
- GitHub Actions 上の PR checks 再実行
- runtime agent skill selection smoke の記録

任意チェックで defect が見つかった場合は、影響範囲を release blocker / follow-up に分ける。

### follow-up の扱い

follow-up は、v0.7.0 tag を止めないが、v0.7.x 以降で明示的に再確認する残件として扱う。

分類:

- `security/dependency`: runtime dependency ではない監査環境側 CVE、production blocker ではない moderate / dev dependency audit 残件
- `runtime/profile`: release smoke を止めない既知 warning、local container profile の cleanup
- `validation/noise`: tests や静的解析の noise 削減
- `optional/coverage`: clean clone、CI rerun、DB integration、browser e2e など確認範囲の追加

follow-up 記録時に残す項目:

- 検出元
- 現象
- v0.7.0 blocker にしない理由
- 推奨対応時期
- 次回確認コマンドまたは確認観点

## 6. 証跡の残し方

release 前確認の結果は、次のいずれかで残す。

- PR comment
- `docs/releases/KOIKI-FW_0.7.0.md` の Validation 欄
- 必要に応じて `docs/dev/v0.7.0-final-release-check-results.ja.md`

記録する項目:

- 対象 commit SHA
- 実行日
- 実行環境
- 実行コマンド
- 成功 / 失敗
- warning / known issue
- release blocker の有無
- tag 作成判断

## 7. tag / GitHub release 判断

tag は topic branch では切らない。

想定手順:

1. final check plan / docs を `dev/v0.7` へ反映する
2. release candidate commit を明確にする
3. final check を実行する
4. release blocker がなければ、release 対象 branch / commit から `v0.7.0` tag を作成する
5. `docs/releases/KOIKI-FW_0.7.0.md` を元に GitHub release を作成する

tag 作成前の判断:

- mandatory check がすべて成功している
- known issue が release note または PR comment に記録されている
- Todo API sample router 維持判断に未解決の矛盾がない
- `app.main:app` 互換維持が release note に記録されている
- release note が tag 対象 commit の実態と一致している
