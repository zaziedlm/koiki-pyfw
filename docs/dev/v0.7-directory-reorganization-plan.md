# KOIKI-FW v0.7 Directory Reorganization Plan

## Purpose

This document defines a concrete repository reorganization plan for the `v0.7` line.

The main objective is to make KOIKI-FW easier to provide to other projects in two forms:

1. as a reusable framework library
2. as a reference application that composes that framework

The current repository already separates `libkoiki/` and `app/`, but packaging, startup, migrations, and tests are still partially mixed. This plan makes those delivery boundaries explicit.

## Document Synchronization Rule

This English plan and the Japanese plan are a synchronized pair.

- `v0.7-directory-reorganization-plan.md`
- `v0.7-directory-reorganization-plan.ja.md`

When one of these files is updated, the other must be reviewed and updated as needed in the same change window.

At minimum, keep the following aligned across both documents:

- scope and non-goals
- branch assumptions and baseline branch names
- roadmap stages and deliverables
- naming and layout decisions

## Target Outcome

By the end of the reorganization:

- `libkoiki` is a clean publishable framework package
- the current `app/` becomes a named reference application package
- project-specific applications have a separate top-level home under `apps/`
- `frontend/` remains at repository root as the starter frontend paired with the backend template
- the repository root becomes a workspace for development, CI, docs, and local operations
- runtime startup is based on an app factory instead of a single hard-wired `app.main`
- component-local migrations, tests, and docs are easier to understand and maintain

## Current Structural Problems

The current repository has these practical issues:

- the root `pyproject.toml` is a workspace-oriented file, not a clean publishable package definition
- `app/` is a generic name and is not even a regular Python package yet
- `libkoiki` includes both framework concerns and concrete sample/reference API behavior
- startup composition is centered in `app/main.py`, which makes reuse by external projects harder
- ORM extension points rely on import side effects
- tests are present, but package ownership is not always obvious from the layout

## Ownership Rules In The Target Layout

### `components/libkoiki`

This package owns:

- reusable framework behavior
- shared auth, logging, middleware, rate limiting, monitoring
- reusable DB/session abstractions
- reusable schemas, repositories, services, and extension contracts
- app factory primitives and composition hooks that other projects can consume

This package should not own:

- project-specific SSO providers
- business-specific API endpoints
- sample-domain CRUD that is not essential to the framework contract
- reference-app startup details

### `components/koiki_ref_app`

This package owns:

- the current application-specific API behavior
- current SSO and SAML integrations
- reference application startup
- project-specific models and migrations
- reference application configuration defaults

This package is the canonical example of how to build on top of `libkoiki`.

### `apps/`

This top-level area owns:

- project-specific applications
- customer-specific or deployment-specific backend implementations
- downstream application code that should not automatically be treated as reusable upstream assets

This area should not become the default home for reusable framework or template code.

### Repository Root

The root owns:

- workspace tooling
- CI
- Docker and local operations
- the paired `frontend/` starter implementation
- end-to-end tests spanning multiple components and applications
- repository-wide docs

The root should not pretend to be the main runtime package distributed to users.

### Framework component

Current:

- `libkoiki/`

Target:

- `components/libkoiki/src/libkoiki/`

Keep almost all existing framework code, but move it under `src/` packaging.

### Reference application component

Current:

- `app/`
- `app/main.py`
- `app/core/`
- `app/api/v1/`
- `app/models/`
- `app/repositories/`
- `app/services/`
- `app/schemas/`

- `components/koiki_ref_app/src/koiki_ref_app/`

### Migrations

- root `alembic/`
- root `alembic.ini`

Target:

- `components/koiki_ref_app/alembic/`
- `components/koiki_ref_app/alembic.ini`

Reason:

- the current migrations are tied to the concrete composed application schema, not the abstract framework alone

### Entry points

Current:

- root `main.py`
- `app/main.py`

Target:

- `components/koiki_ref_app/src/koiki_ref_app/asgi.py`
- `components/koiki_ref_app/src/koiki_ref_app/app_factory.py`

Optional compatibility wrappers can remain at the root during migration, but they should stop being the canonical startup target.

### Tests

Current:

- `tests/unit/app/`
- `tests/unit/libkoiki/`
- `tests/integration/app/`
- `tests/integration/services/`
- `tests/e2e/`

Target:

- `components/libkoiki/tests/unit/`
- `components/libkoiki/tests/integration/`
- `components/koiki_ref_app/tests/unit/`
- `components/koiki_ref_app/tests/integration/`
- root `tests/e2e/`

Reason:

- ownership becomes obvious
- component-local CI can run in smaller scopes
- reusable framework tests no longer depend on app layout assumptions

### Frontend

Current:

- `frontend/`

Target:

- keep `frontend/` at repository root

Reason:

- the frontend has a different toolchain and runtime model from the Python components
- keeping it at root preserves visibility for AI agents and human contributors
- the frontend is paired with the backend template, but is not part of the `libkoiki` package itself

### Project-specific applications

Current:

- no dedicated top-level area exists for project-specific applications

Target:

- `apps/`

Reason:

- `components/` should stay focused on reusable framework, reference, and platform components
- `apps/` provides a clear home for project-specific application implementations
- this keeps the distinction between reusable upstream assets and downstream application code visible to both humans and AI agents

## Required Internal Refactoring

Directory moves alone are not enough. The following code-level reorganizations are part of the plan.

### 1. Introduce explicit app factory composition

Add framework-side composition primitives such as:

- `libkoiki.app_factory.create_base_app(...)`
- `libkoiki.app_factory.install_framework_middlewares(...)`
- `libkoiki.app_factory.include_framework_routers(...)`

Then let the reference app do:

1. build the base app from `libkoiki`
2. register reference-app config and integrations
3. include reference-app routers
4. install reference-app background tasks

This is preferable to one concrete `app.main` owning everything.

### 2. Replace ORM side-effect binding with explicit bootstrap

Current pattern to avoid long-term:

- importing `app.models` mutates `libkoiki.models.UserModel`

Recommended target:

- add `koiki_ref_app.bootstrap.orm` module
- define explicit functions such as `register_model_extensions()`
- call bootstrap during app creation and migration startup

The goal is to eliminate hidden import-order requirements.

### 3. Split framework APIs from sample/reference APIs

Recommended split inside `libkoiki`:

- keep auth, user, permission, and framework-monitoring APIs only if they are true framework contracts
- move `todos` and similar sample-domain endpoints into:
  - `components/koiki_ref_app`, or
  - `examples/`

If an API exists only to demonstrate the framework, it should not be part of the default framework surface.

### 4. Standardize package metadata

For `libkoiki`:

- keep `pyproject.toml`
- remove legacy `setup.py`
- use one authoritative dependency definition

For the reference app:

- define a real publishable package
- move test-only dependencies out of runtime dependencies
- declare `libkoiki` as a normal dependency

For the root workspace:

- keep a lightweight workspace `pyproject.toml`
- avoid presenting it as the main distributable application package

## Recommended Delivery Order

The work should not start with large directory moves.

The safest order for this repository is:

1. stabilize the branch model for `v0.6` maintenance and `v0.7` work
2. clean up package metadata and dependency ownership
3. migrate dependency management from Poetry to `uv`
4. introduce app-factory composition without moving major directories yet
5. define the template distribution model explicitly
6. move to `components/` and `src/`, and reserve `apps/` for project-specific applications
7. move migrations, scripts, and CI to the new shape
8. remove temporary compatibility shims

This order reduces risk because:

- branch confusion is resolved before structural work starts
- `uv` migration happens before component movement multiplies path changes
- app-factory extraction happens before imports are rewritten
- template decisions are made before packaging and docs are finalized

### Recommended release checkpoints

- create a `v0.6.x` stabilization tag before starting invasive `v0.7` work
- use `release/0.7` only when the new structure is functionally complete and release hardening begins

## `uv` Migration Position In The Plan

The `uv` migration should happen early, but not as the very first change.

Recommended rule:

- first clarify package ownership and remove metadata ambiguity
- then migrate dependency management to `uv`
- only after that move components into their final `components/` layout

This is the preferred order because the current repository still has:

- a root workspace-style `pyproject.toml`
- a separate `libkoiki/pyproject.toml`
- a legacy `libkoiki/setup.py`
- an `app/pyproject.toml` with runtime and test dependency ambiguity

If the repository is moved into a multi-component workspace layout before dependency management is normalized, the migration cost rises unnecessarily.

### `uv` migration goals

For `v0.7`, the `uv` migration should result in:

- `uv.lock` as the lockfile of record
- `uv sync` and `uv run` as the standard local development commands
- root workspace configuration aligned to the final multi-component structure
- component-local dependency definitions kept in `pyproject.toml`
- removal of Poetry-specific workflow assumptions from CI and local docs

### `uv` migration scope before the large move

Do these before moving to `components/`:

- remove `libkoiki/setup.py`
- simplify root `pyproject.toml` to workspace intent
- normalize `libkoiki/pyproject.toml`
- normalize the future reference-app package metadata
- replace Poetry-oriented CI and developer commands with `uv`

## Template Strategy Position In The Plan

Template design should be decided before the final component move is completed.

The recommended delivery model is:

- `libkoiki`
  - reusable framework package
- `koiki_ref_app`
  - reference implementation and starter template for new projects
- `frontend/`
  - root-level starter frontend paired with `koiki_ref_app`

This means the future consumer workflow is not just:

- install `libkoiki`

It also includes:

- start from `koiki_ref_app` when a project wants local code visibility, AI-agent-friendly structure, and project starter assets
- adopt `frontend/` together with `koiki_ref_app` when a project needs the standard UI/BFF starter

### Template goals

The template package should preserve:

- clear local visibility of framework composition
- bundled agent guidance and task-routing docs
- example startup, router composition, and extension points
- starter migrations, tests, and operational scripts where appropriate

The overall template bundle should preserve:

- `koiki_ref_app` as the backend starter
- `frontend/` as the root-level frontend starter
- the agent guidance needed to let AI coding agents work effectively in both layers

The template package should not become:

- a second copy of unrelated historical project baggage
- the new home of framework-only logic that belongs in `libkoiki`

### Template update model

For `v0.7`, template consumption is copy-first.

This means:

- downstream projects are expected to copy/adopt the template structure as a starting point
- downstream projects are not required to remain in a live shared workspace with the upstream repository
- upstream improvements should still be documented and packaged so they can be manually re-applied later

This plan does not require live MCP-backed synchronization between upstream and downstream projects.

## Agent Guidance Strategy

The repository should be restructured to support three classes of AI-agent guidance.

### Skill classes

- maintainer
  - for contributors changing `libkoiki` and the upstream template itself
- template
  - for contributors working inside `koiki_ref_app` and the paired `frontend/` starter
- consumer
  - for downstream projects that adopt the template or integrate `libkoiki`

### Rollout approach

The skill split does not need to be completed all at once.

Recommended rollout:

1. keep the current shared guidance working during structural migration
2. introduce maintainer and template distinction first
3. introduce consumer-oriented guidance after the new template boundaries stabilize

This incremental approach is intentional so the team can measure usefulness and avoid over-design early.

### Skill migration phases

The skill reorganization should happen in three phases.

#### Phase A: Preserve current skills and clarify role intent

In this phase:

- keep the current skills working as-is
- do not force immediate renaming
- annotate each current skill with its future role alignment

Current skills in this phase are interpreted as:

- `koiki-project-overview`
  - temporary cross-repository overview and routing skill
- `koiki-libkoiki-feature-work`
  - effectively the first maintainer-facing framework skill
- `koiki-app-feature-work`
  - effectively the first template-backend skill
- `koiki-auth-security`
  - cross-cutting skill retained across all later phases
- `koiki-testing`
  - cross-cutting skill retained across all later phases

The goal of Phase A is continuity, not taxonomy perfection.

#### Phase B: Split maintainer and template skills

This is the first major structural split.

Add or extract explicit skills for:

- maintainer overview
- maintainer framework work
- maintainer template-backend work
- maintainer template-frontend work
- template overview
- template backend feature work
- template frontend feature work

This phase is the first point where the root-level `frontend/` starter must become explicit in the skill model.

The main reason is that template work is not backend-only:

- contributors will need to understand `koiki_ref_app`
- contributors will need to understand the paired `frontend/`
- contributors will need guidance on when a change belongs in `libkoiki` instead of the template

#### Phase C: Add consumer-oriented skills

This phase should happen only after the new component and template boundaries are stable.

Consumer skills should focus on:

- how downstream projects adopt `libkoiki`
- how downstream projects adopt or copy the template
- how to decide whether a downstream change should remain local or be proposed upstream
- how to understand upgrade and compatibility expectations

Consumer skills are intentionally delayed because their guidance should be based on a stable upstream structure, not a moving target.

### Compatibility policy during the transition

During the migration:

- do not delete legacy skills early
- keep old entry points working while new skills are introduced
- mark old skills as transitional when a preferred replacement exists
- document the mapping from old skill names to new categories

This policy is important because abrupt skill replacement would create unnecessary confusion for both human contributors and AI agents.

### Initial target structure for skill families

The expected long-term shape is:

- maintainer
  - overview
  - libkoiki
  - template-backend
  - template-frontend
  - auth-security
  - testing
- template
  - overview
  - backend-feature
  - frontend-feature
  - auth-integration
  - testing
- consumer
  - overview
  - libkoiki-integration
  - auth-setup
  - upgrade-guide

Not every skill in this target structure must exist immediately.

The priority order is:

1. preserve the current shared skill set
2. establish maintainer and template distinction
3. add consumer guidance after the template model proves stable

## MCP Position In The Plan

MCP enablement is explicitly out of scope for the current `v0.7` reorganization.

Reason:

- the repository still needs structural and packaging clarity first
- skills and local guidance can provide immediate value without adding MCP design work
- MCP should be reconsidered only after the template and guidance structure has stabilized

Therefore:

- do not block `v0.7` on MCP design
- do not couple the current migration phases to MCP rollout
- treat MCP as a follow-up initiative after the repository structure, template model, and skills model are proven

## Migration Phases

### Phase 0: Freeze the target shape

Before moving files:

- approve this directory target
- decide final package name for the reference app
- decide which APIs remain part of `libkoiki`
- decide whether `todos` is framework, example, or reference-app behavior

### Phase 1: Packaging cleanup without large moves

Do first:

- remove `libkoiki/setup.py`
- make `app/` a real package or rename target package in a new location
- clean up runtime vs dev dependencies in `app/pyproject.toml`
- define root workspace intent clearly

This phase reduces ambiguity before large file movement.

### Phase 2: Introduce app factory

Do next:

- create framework-side app factory helpers
- create reference-app `create_app()`
- move startup logic from current `app/main.py` into explicit bootstrap modules
- keep compatibility wrappers so existing commands still work

This phase reduces risk before path migration.

### Phase 3: Move to `components/` + `src/`, and introduce `apps/`

After factory introduction:

- move `libkoiki/` to `components/libkoiki/src/libkoiki/`
- move `app/` to `components/koiki_ref_app/src/koiki_ref_app/`
- introduce `apps/` as the reserved top-level home for project-specific applications
- move component-local tests under each component
- update imports, tooling, and Docker paths

### Phase 4: Move migrations and scripts

Then:

- move Alembic files into the reference app component
- update migration commands and CI
- update docker entrypoints and local scripts

### Phase 5: Remove transitional compatibility

Finally:

- remove root runtime wrappers if no longer needed
- remove import shims
- remove any remaining side-effect model bootstrap

Once the layout changes:

- CI should test `components/libkoiki` and `components/koiki_ref_app` separately
- root E2E should run only after component-level checks pass
- coverage should be reported per component, then optionally aggregated
- Docker build context should stay at root, but startup module should point to the reference app package
